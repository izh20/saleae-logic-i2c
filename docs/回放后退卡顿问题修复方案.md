# 回放后退卡顿问题修复方案

## 问题描述

回放中途按空格暂停后再按左键（后退），UI 会无响应，因为每次后退一帧都要重放 0~N 帧的所有数据。

**日志证据：**
```
[usePlayer] stepBackward: {currentFrameIndex: 1774, prevIndex: 1773}
[usePlayer] seek called: {index: 1773, isBackward: true, currentFrameIndex: 1774, maxIndex: 3795}
[usePlayer] rebuildTrajectories from 0 to 1773  // 要同步处理 1773 帧！
```

**问题定位：**
- `rebuildTrajectories` 使用同步循环处理所有历史帧
- 帧数越多，阻塞时间越长
- 3795 帧时，每按一次左键需要处理近 2000 帧

## 修复方案

### 方案 A：只显示当前位置

**思路：** 后退时不重建历史轨迹，只显示当前帧的位置信息。

**改动：**
1. `seek` 后退时不再调用 `rebuildTrajectories`
2. 直接显示目标帧的位置
3. 前进时正常累积轨迹

**效果：**
| 操作 | 轨迹显示 |
|------|----------|
| 前进 | 完整轨迹（累积） |
| 后退 | 只显示当前位置，无历史轨迹 |
| 再前进 | 从当前位置继续累积 |

**优点：**
- 无阻塞，响应立即
- 实现简单

**缺点：**
- 后退时看不到手指走过的路径

---

### 方案 B：异步分片重建轨迹

**思路：** 将同步重建改为分片异步执行，每帧只处理少量帧，避免阻塞。

**改动：**
1. `rebuildTrajectories` 改用 `requestIdleCallback` 或分批 `setTimeout`
2. 每批处理 100 帧，分片完成
3. 期间 UI 可响应其他操作

**效果：**
| 操作 | 轨迹显示 |
|------|----------|
| 前进 | 完整轨迹 |
| 后退 | 轨迹逐步重建，显示进度 |

**优点：**
- 轨迹完整
- UI 不完全阻塞

**缺点：**
- 实现复杂
- 后退时轨迹重建有延迟

---

### 方案 C：轨迹缓存优化

**思路：** 在内存中缓存每个关键帧的轨迹状态，后退时直接加载缓存而非重放。

**改动：**
1. 每 100 帧保存一个轨迹快照
2. 后退时找到最近的关键帧快照
3. 从快照继续重放到目标帧

**效果：**
| 操作 | 轨迹显示 |
|------|----------|
| 前进 | 完整轨迹 |
| 后退 | 快速恢复轨迹（从快照加载） |

**优点：**
- 轨迹完整
- 性能较好

**缺点：**
- 实现复杂
- 增加内存占用

---

### 方案 D：增量撤销栈（推荐）

**思路：** 前进时记录每帧对轨迹的增量修改（diff），后退时 pop 并反向操作，O(1) 恢复上一帧状态。

**核心观察：** `handleFrame` 对轨迹的操作有四种：
- **finger touch**：往某个 finger 的 `points[]` 追加一个点
- **finger release**：删除该 finger 的整条轨迹
- **stylus touch/hover**：往 stylus 轨迹追加一个点
- **stylus release**：清空 stylus 轨迹

每帧 diff 极其轻量，只需记录"追加了什么"和"删除了什么"。

**数据结构：**
```ts
interface FrameDiff {
  // finger release 事件：删除的轨迹（恢复时原样放回）
  deletedFingerTrajectories: Map<number, FingerTrajectory>;
  // finger touch 事件：追加了点的 fingerId 列表（撤销时 pop 最后一个点，若 pop 后 points 为空则删除该轨迹）
  appendedFingerIds: number[];
  // stylus touch/hover 事件：追加前的点数量（撤销时截断到该数量）
  stylusPointsBefore: number;
  // stylus release 事件：删除前的完整 stylus 轨迹（恢复时原样放回）
  deletedStylusTrajectory: { x: number; y: number; state: StylusState }[] | null;
}
```

**快照机制（用于大幅跳转）：**
- 每 **200 帧**保存一个完整轨迹快照
- 快照结构：`{ index: number, trajectories: Map, stylusTrajectory: [] }`
- 大幅跳转时：找最近快照 + 重放 ≤200 帧

**流程：**

| 操作 | 执行逻辑 | 复杂度 |
|------|----------|--------|
| 前进一帧 | 正常 handleFrame + push diff 到 undoStack | O(1) |
| 后退一帧 | pop diff，反向操作（删点 / 恢复轨迹） | O(1) |
| 前进跳转 | 找最近快照 → 重放 ≤200 帧，重建 undoStack | O(200) |
| 后退跳转 | 找最近快照 → 重放 ≤200 帧，重建 undoStack | O(200) |

**关键边界处理 — seek 跳转时的 undoStack 一致性：**

不论前进还是后退跳转，如果跨度 > 1 帧，undoStack 与帧索引会错位。统一处理：
1. 找到目标帧之前最近的快照
2. 恢复快照的轨迹状态
3. 清空 undoStack
4. 从快照帧逐帧重放到目标帧，**每帧都构建 diff 并 push 到 undoStack**
5. 这样跳转后 undoStack 仍然可用，后续单步后退正常工作

**改动：**
1. `TrajectoryView` / `PlaybackView` 中增加：
   - `undoStackRef: FrameDiff[]`
   - `snapshotInterval = 200`，`snapshotRef` 保存快照
2. `handleFrame` 中在修改轨迹的同时构建 diff 并 push
3. 新增 `undoFrame()` 方法：pop diff → 反向操作（pop 后 `points` 为空时需删除该轨迹条目）
4. 新增 `takeSnapshot()` 和 `restoreSnapshot()` 方法
5. `usePlayer.seek`：
   - 单步后退 → 调用 `undoFrame()`
   - 跳转（前进/后退跨度 > 1）→ 快照恢复 + 逐帧重放并重建 undoStack

**效果：**
| 操作 | 轨迹显示 |
|------|----------|
| 前进 | 完整轨迹（累积） |
| 后退 | 完整轨迹（逐帧撤销） |
| 大幅跳转 | 完整轨迹（从快照恢复） |

**优点：**
- 单步后退 O(1)，无任何 UI 阻塞
- 轨迹完全保留，用户体验无损
- 内存开销极低（每帧 diff ~50-100 bytes，3795 帧 ≈ 200-400 KB）
- 大幅跳转只需 O(200)，可接受

**缺点：**
- 实现复杂度中等（需修改 handleFrame 和 seek 逻辑）

## 方案对比

| | 方案 A | 方案 B | 方案 C | **方案 D** |
|--|--------|--------|--------|-----------|
| 单步后退复杂度 | O(1) | O(N) | O(N/K) | **O(1)** |
| 大幅跳转复杂度 | O(N) | O(N) | O(K) | **O(200)** |
| 轨迹完整 | ❌ | ✅ | ✅ | **✅** |
| 额外内存 | 无 | 无 | 高 (快照) | **低 (diff)** |
| UI 阻塞 | 无 | 有延迟 | 短暂 | **无** |
| 实现难度 | 低 | 中 | 中高 | **中** |

## 推荐

**推荐方案 D（增量撤销栈）**，理由：
1. 单步后退 O(1)，响应即时
2. 轨迹完全保留，体验无损
3. 内存开销极低，远优于方案 C 的全量快照
4. 改动集中在 `handleFingerFrame` 和 `seek` 两处，不需要重构架构

如果后续需要支持拖动进度条大幅跳转，可叠加方案 C（每 200 帧存一个快照），最坏只需重放 200 帧。
